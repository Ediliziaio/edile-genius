

## Security Audit & Hardening Plan

### Findings Summary

| # | Risk | Severity | Location |
|---|------|----------|----------|
| 1 | `.env` not in `.gitignore` — committed to repo | **Low** | `.gitignore` (values are public keys, but bad practice) |
| 2 | No `.env.example` for developer onboarding | **Low** | Missing file |
| 3 | `el_api_key` (ElevenLabs) stored in `companies` table, read/written from client code | **High** | `Settings.tsx`, `CompanyDetail.tsx`, `CreateCompany.tsx` |
| 4 | `supabase/client.ts` hardcodes URL + anon key | **None** | Auto-generated by Lovable; these are publishable keys — acceptable |
| 5 | Edge functions use `Deno.env.get()` for all real secrets | **None** | Correct pattern already in place |
| 6 | No secrets leaked in console.log statements | **None** | Verified across all 26 edge functions |

### Critical Issue: `el_api_key` in Database

The `companies` table has an `el_api_key` column that stores ElevenLabs API keys as plaintext. Client-side code in `Settings.tsx` and `CompanyDetail.tsx` reads and writes this column directly via the Supabase JS client. Even with RLS, this means the API key travels over the wire to the browser.

However, the platform already migrated to a centralized `ELEVENLABS_API_KEY` Supabase Secret used by all edge functions. The per-company `el_api_key` column appears to be **legacy/unused by backend logic** — edge functions read `Deno.env.get("ELEVENLABS_API_KEY")`, not the DB column.

### Plan

**1. `.gitignore` — add `.env` pattern**
- Add `.env` and `.env.*` (except `.env.example`) to `.gitignore`
- The current `.env` only has public `VITE_` vars so no emergency, but it prevents future accidents

**2. Create `.env.example`**
- Document the 3 required `VITE_` variables with placeholder values
- Add comments clarifying these are public/publishable keys

**3. Deprecate `el_api_key` from client-side access**
- In `Settings.tsx`: remove the API key input/save section (the platform uses centralized secrets now)
- In `CompanyDetail.tsx`: remove the editable API key field, show only a status badge based on `platform_config.el_api_key_configured`
- In `CreateCompany.tsx`: remove `el_api_key` from the creation form
- This eliminates the only real secret exposure risk in the codebase

**4. No other changes needed**
- Edge functions already use `Deno.env.get()` correctly
- No service role key in client code
- No secrets in console.log output
- `supabase/client.ts` values are publishable by design

### Files Modified
1. `.gitignore` — add `.env` exclusion
2. `.env.example` — new file with documented placeholders
3. `src/pages/app/Settings.tsx` — remove ElevenLabs API key section
4. `src/pages/superadmin/CompanyDetail.tsx` — remove editable API key field
5. `src/pages/superadmin/CreateCompany.tsx` — remove `el_api_key` from form

### Key Rotation Advisory
- The Supabase anon key in `.env` and `client.ts` is **publishable** — no rotation needed
- The `ELEVENLABS_API_KEY` Supabase Secret is properly secured — no rotation needed
- If any company had a per-company `el_api_key` stored in the DB, those keys should be rotated by the company after we remove client access, as they were exposed to browser clients

### No DB Migration Required
- The `el_api_key` column can stay in the DB (backward compatible) — we just stop exposing it client-side
- Optionally a future migration could drop the column, but that's non-urgent
